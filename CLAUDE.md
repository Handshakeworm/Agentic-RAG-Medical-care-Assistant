# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

**This is a spec-driven project in the skeleton stage.** Most files under `src/` are still empty placeholders (verify with `wc -l <file>` before assuming an import works). Real content lives in:
- `DEV_SPEC.md` (~4600 lines) — the **single source of truth** for architecture, schemas, and acceptance criteria.
- Modules already implemented are tracked in DEV_SPEC §8.4 progress table — tasks marked `[x]` correspond to non-empty production modules; `[~]` tasks may have partial code. **Never enumerate "implemented modules" here**; check §8.4.
- `.claude/skills/auto-coder/` — the workflow that turns DEV_SPEC into code.

Implementation proceeds task-by-task through the `auto-coder` skill (triggered by "auto code" / "自动开发"). It reads the §8.4 progress table in `DEV_SPEC.md`, picks the next `[ ]` task, implements per spec, runs tests, and commits.

## Authority hierarchy (read this before resolving any conflict)

1. `DEV_SPEC.md` is authoritative. **Never** edit `.claude/skills/auto-coder/references/*.md` — they are regenerated by `sync_spec.py` and any manual edits are silently overwritten.
2. Inside DEV_SPEC: **§9 (global contracts) > business chapters (§3/§4/§6) > §8 (schedule)**. The schedule is an execution guide, not a design decision.
3. Inside §9: the canonical lists win — §9.3 (full LLM-call inventory), §9.5 (full Pydantic schemas), §9.7 (`agent_limits` constants). A call site / field / constant not listed here is a violation, not an oversight.
4. When DEV_SPEC and existing code disagree, **stop and ask the user** — do not silently pick a side.

## Spec citation discipline (also for interactive coding, not just auto-coder)

When implementing or modifying any non-trivial behavior:

1. **Locate the spec section first.** Before writing code that touches an Agent node, RAG stage, schema, or runtime constant, find the authoritative spec section (use `grep -n "^### " DEV_SPEC.md` to navigate). If no spec section exists, surface this gap to the user *before* writing code.

2. **Cite the section in code.** File-level docstrings, key function docstrings, and non-trivial test names should reference the spec section they implement (e.g. "DEV_SPEC §9.7" / `test_..._match_spec_9_7_1`). This survives PR rebase and lets future readers map code → spec without grep.

3. **Surface gaps and conflicts proactively.** If you notice a spec/code mismatch (even one tangential to the current task), raise it as a sidebar in the same response — do not silently fix or silently ignore.

4. **No silent re-derivation.** If a constant, schema field, or LLM call appears in §9.3/§9.5/§9.7's canonical lists, link to that list in the citing comment instead of re-stating the value — single source.

Authority hierarchy already declares §9 wins over business chapters, but in practice it's easy to read §3/§4/§6 (the chapter that "feels relevant") and forget §9 even exists. Before writing code, sweep the §9 subsections that match what you're about to touch:

| Implementation | Sweep these §9 subsections first |

## High-level architecture

**Single-process API container** (`api`) that bundles FastAPI + LangGraph Agent + RAG pipeline + Embedding + Reranker. LLM inference is **cloud-side** (DashScope OpenAI-compatible API) so the entire 16GB GPU is reserved for Embedding (Qwen3-Embedding-8B INT8, ~8.5GB) and Reranker (BGE-Reranker-v2-minicpm-layerwise INT8, ~2.6GB) — they share the GPU because their loads are naturally offset (Embedding at ingest, Reranker at query).

```
nginx ──► api (FastAPI + Agent + RAG + GPU models) ──► DashScope (LLM)
                              │
                              ├──► Milvus standalone (+ etcd, minio)   vectors
                              ├──► PostgreSQL                          metadata + raw_documents + audit (rag_trace)
                              └──► Redis                               config cache ONLY (see §5.1)
```

### Agent workflow (`src/agent/`)

A LangGraph `StateGraph` with **16 nodes + 2 conditional routers**. The shared `MedicalState` (TypedDict) is defined in `src/agent/state.py`. Two interrupt-driven loops sit inside the main flow:

```
① info_collect → ①.5 analyze_initial_reports → ② build_query → ③ retrieve →
④ extract_symptoms → ⑤ select_discriminative_symptom →
  [router: should_continue]
    ├─ followup loop:    ⑥a generate_followup → ⑥b wait_followup_answer (interrupt) → ⑦ process_followup → ②
    └─ ⑩ diagnose →
       [router: diagnose_router]
         ├─ exam loop:   ⑧a recommend_exam → ⑧b wait_exam_report (interrupt) → ⑨ process_exam_result → ②
         └─ ⑪ safety_gate → ⑫ generate_advice → ⑬ format_response → END
```

Notes on this graph that are easy to get wrong:
- ⑥/⑧ are intentionally split into `a` (LLM generates question) and `b` (`interrupt()` waits for user) — keeping them separate prevents the LLM call from re-firing on resume.
- ⑩ `diagnose` is a **three-step LLM chain** (`EvidenceSheet` → `DiagnosisRanking` → `DiagnosisOutput`). If any step fails after 3 retries, the whole chain short-circuits to an `insufficient` result and writes `failure_reason="step_{N}_structured_output_failed: ..."`. **Never** feed a partial/empty intermediate to the next step.
- ⑩ also has a **Step -1** that short-circuits before any LLM call when `state["followup_round"] >= settings.agent_limits.MAX_FOLLOWUP_ROUNDS` (writes `failure_reason="followup_round_capped"`).
- `should_continue` router is a **pure function** — it must not mutate state. Cap-handling lives in ⑩ Step -1, not the router.
- `present_illness_slots` has exactly **13 dimension slots**; multi-value slots (`aggravating`, `relieving`, `associated_symptoms`) are `list[str]`, all others are `str | None`.

### RAG pipeline (`src/rag/`)

**Ingestion** (`ingestion/`): MinerU output → parent/child chunking (parents skip vectorization, written to `chunks` with `embedding_status='skip'`; children get vectorized; tables produce both whole-table summary chunks and per-row chunks) → LLM enrichment (title/summary/tags/hypothetical_questions) → multi-vector Embedding (1 original + 1 summary + 2~3 question per chunk) → Milvus + PostgreSQL with idempotent upsert + zombie cleanup.

**Retrieval** (`retrieval/`): Dense (Qwen3-Embedding-8B → Milvus ANN, single shot) + Sparse (Milvus BM25, **one query per dimension** in `sparse_queries`) → single-stage multi-route RRF → dedup by `source_chunk_id` → Top-N. The Reranker is **not** invoked during retrieval — it runs as Step 0 of node ⑩ `diagnose`, with a fallback to RRF order on timeout/failure.

### Storage layout

- **Milvus 2.5.9 standalone** (pinned to `pymilvus>=2.5.0,<2.6` in `pyproject.toml`). Two collections:
  - `docs_collection` — chunk vectors with `vector_type` field (`original` / `summary` / `question`); only `original` records carry the BM25 sparse vector.
  - `terms_collection` (defined in `config/milvus_schema.py`) — alias-level term vectors (4096-dim) for entity linking. PK is `{concept_id}_{alias_index}`.
- **PostgreSQL 16** — `sources`, `raw_documents` (MinerU outputs as JSONB + GIN), `chunks`, `users`, `patients`, `conversations`, `system_config`, plus audit tables: `rag_trace` (90-day retention), `kb_change_log`, `config_change_log`, `diagnosis_feedback`.
- **Redis 7** — **config cache only** (60s TTL). RAG response caching is intentionally not implemented (Agentic state mutates between calls; query string would collide across patients/rounds — see `pending tasks.md` and §5.1).

### `config/settings.py` and `agent_limits` (§9.7)

All runtime caps and thresholds are centralized in `AgentLimitsSettings` (`pydantic_settings.BaseSettings`, env prefix `AGENT_`). The seven authoritative constants:

| Constant | Default | |
|---|---|---|
| `MAX_FOLLOWUP_ROUNDS` | 8 | followup hard cap |
| `MAX_EXAM_ROUNDS` | 3 | exam loop hard cap |
| `MAX_FOLLOWUP_QUESTIONS` | 5 | per-round question quota (dimension + symptom combined) |
| `RETRIEVE_TOP_N` | 200 | post-RRF truncation |
| `ASKABLE_GAIN_THRESHOLD` | 0.15 | symptom-level info-gain floor |
| `ENTITY_LINKING_TIER2_THRESHOLD` | 0.92 | terms_collection cosine cutoff |
| `RERANKER_CUTOFF_LAYERS` | None (= full 28) | layerwise early-exit |

**Hard rule**: business code must `from config.settings import settings` and read `settings.agent_limits.X`. Module-level constants like `MAX_X = 8` in node files are violations — they fragment under multi-task implementation. Tune via `.env` (e.g., `AGENT_MAX_FOLLOWUP_ROUNDS=10`), never code edits.

## LLM call contract (§9.1) — the most-violated rule

**No decorators. No helper functions. No context managers.** Every LLM call site writes its own ~10–15 lines of `try/except/finally` boilerplate. The ~300 lines of duplication across 20+ call sites is **intentional** — it keeps exception scope clear, makes `with_retry`'s internal behavior visible, and avoids signature constraints.

Each call site must:
1. Build the chain inline: `chain = llm.with_structured_output(Schema).with_retry(stop_after_attempt=N)`.
2. Wrap `chain.invoke(prompt, config={"callbacks": [retry_observer], "metadata": {"node": ..., "schema": ...}})` in `try/except/finally`.
3. Manually report 6 Prometheus metrics with the right labels (see §9.1 table). `_attempts.inc()` before the try; `_failures.inc()` in `except`; `_fallbacks.inc()` before the fallback path; `_latency.observe()` in `finally`. Retries are auto-reported by `RetryObserver` (a `BaseCallbackHandler` subclass declared in `src/common/metrics.py`).
4. Choose a fallback per the table in §9.3:
   - **High** safety (⑩ diagnose, ⑪ safety_gate): full-chain bailout to `insufficient` / conservative warning.
   - **Mid** safety (most agent nodes): raise after retries → graph terminates the session.
   - **Low** safety (3.1.3 enrichment, evaluation judges): skip the chunk / mark `eval_failed`.

`src/common/metrics.py` is **declaration only**: 6 module-level singleton `Histogram`/`Counter` objects + the `RetryObserver` class. **No** wrapper functions, full stop.

### Schema evolution (§9.2)

`MedicalState` and `rag_trace.retrieved_chunks` are reread from checkpointers and JSONB long after they're written. New fields **must** carry a default (`Field(None, ...)` or `Field(default_factory=...)`). Type widenings are fine (`str` → `str | None`); narrowings break old data. `Literal[...]` enums may only **gain** values, never lose them. Breaking changes require a `SchemaV2` shipped alongside V1 for at least one retention window.

**All test files live under `tests/{unit,integration,e2e}/` as `test_*.py`** — including smoke / exploratory / no-assertion checks where the value is "see if this runs without crashing." Never under `scripts/`, which is for operational tools (`batch_parse_pdfs.sh`, `init_db.py`, etc.) — not verification. If a test needs heavy resources (GPU model load, real API calls), gate it with `pytestmark = pytest.mark.skipif(...)` instead of moving it out of `tests/`.

`pytest-asyncio` is included; async tests need the appropriate marker. Integration tests require `docker compose up -d` first.

## When auto-coder fails or stops

The skill stops and asks instead of guessing in three situations:
1. **Test failure that isn't a clear code bug** — could be a test/spec/environment bug. Don't auto-edit the test or DEV_SPEC; classify with the user first.
2. **Spec internal conflict** — e.g. §4.1.2 says 13 fields but §9.5 says 12. The skill never silently picks a side.
3. **Missing prerequisite** — a dependent task in §8.4 is `[ ]`. The skill asks before forging ahead.

If you must continue past one of these, the user has to give explicit override.
