"VLLM_USE_V1 不被这个版本识别（日志显示 Unknown vLLM environment variable），说明 v0.16.1rc1 已经完全移除了 V0 引擎，这个方法失效了。

这个 vLLM 版本（v0.16.1rc1）存在一个 Bug：它的 assert 阻止了 Mamba MoE 模型使用 cpu_offload_gb，但这两个功能在技术上是独立的，互不干扰。
目前唯一可行的不修改源码方案已全部尝试失败。"

2. 控制gpu_memory_utilization=0.7，为运行过程创造临时显存空间（如triton kernel自动调优，需要剩余足够显存）

4. language_model_only=True,   # 跳过视觉编码器的缓存占有，使用纯文本模式（官方推荐）
5. enable_chunked_prefill=False,  # 关闭 chunked1 prefill，避免 Mamba block size 重初始化冲突
6. cpu_offload_gb=16,  # offload 16GB 到 CPU
7. **注意我这没有使用只offload experts的设置，这是一个问题**

#### 结论：
显然不可用于RAG项目和需要快响应的业务场景。虽然包括了首次调优时间的速度，但是后续的推理即使速度更快也好不到哪里去，受限于cpu_offload要使用过低的pcie带宽。更多原因如下：moe需要加载所有参数，我们offload部分参数到了cpu上，但是每个token的生成都由router决定使用哪些experts，因此experts会时常在显存和内存间跑来跑去，导致传输速率仍旧非常影响实际推理。**这里还犯了没有只offload experts层的错误，不过做了实际表现也不会比lamma.cpp好的。**
