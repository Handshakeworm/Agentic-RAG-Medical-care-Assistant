# 第4章 Agent 设计 — 审阅问题清单


### 2. 急危重症缺乏前置检测

当前设计中，高危情况（疑似心梗、脑卒中）的检测和提示在 `generate_advice ⑬` 才发生。但在真实临床场景中，如果患者输入"胸口剧烈疼痛伴大汗、左臂麻木"，经过完整的 ①→②→③→④→⑤→⑥（可能还有多轮追问）→⑪→⑫→⑬ 的流程才给出"立即就医"提示，延迟过长。

**建议**：在 `info_collect ①` 之后增加一个轻量级急危重症筛查节点（或在 ① 内部添加 Step 0），用规则+简单 LLM 判断做红旗症状（red flag symptoms）匹配。命中时直接短路到 `format_response`，输出紧急就医建议，不走后续完整流水线。





缺陷 3（安全风险，**⏸️ 暂搁置待后续完善**，DEV_SPEC.md §4.1.2 Node ⑪ 已加 TODO 注释）：safety_gate ⑪ 声称"确定性规则过滤"但实际使用 RAG
规范写道：规则层通过 RAG 检索对应 chunk 进行匹配判断，同时又将其定性为"规则过滤（确定性）"。

RAG 是概率性检索，若相关 chunk 未被召回，过敏药物/配伍禁忌过滤规则就不会触发。这对于安全关键的用药约束而言是根本性的架构矛盾——声称确定性但实现是概率性的，可能导致已知禁忌药物漏检。

**已讨论的重构方向（2026-04-20，未落地）**：
- 摄取期从《临床用药指南》一次性结构化抽取规则入 PostgreSQL 表（drug_allergy_rules / drug_interaction_rules / drug_pregnancy_categories），查询期改为 SQL 直查 O(1) 命中
- 统一命名源：指南抽取同时产出 terms_collection 中 drug 类型的 preferred_term + concept_id，规则表以同一 concept_id 为外键，保证 Entity Linking 归一化后必匹配（避免 terms_collection 命名与指南命名不一致导致的硬查询漏匹配）
- 药物别名补强：可接入 CFDA 药品通用名-商品名对照表作为 terms_collection 的补充源
- Fail-safe：规则库未覆盖的药物一律标记"待药师复核"而非静默放行
- RAG 退化为 LLM 兜底层（交叉过敏、罕见相互作用、剂量调整）的语义辅助，不再作为规则判定依据

**搁置原因**：改动面较大（摄取管线分支 + DB schema + Node ⑪ 重写 + 评估指标新增），MVP 阶段暂按现设计实现；落地前集成测试必须显式覆盖"禁忌药漏召回"用例，出现一次即触发重构。











# 基础设施

###  1. RAG 响应缓存 key 设计与 Agentic 场景不兼容（中影响）

5.1 节缓存设计：`rag:<hash(query_text)>` 缓存完整 RAG 响应，TTL 1h。

在 Agentic 工作流中，同一个 query 字符串在不同患者、不同追问轮次（不同的 `confirmed_symptoms`/`denied_symptoms`/`report_findings`）下，应召回并推理出不同的结果。用纯 query 字符串做 key，会导致不同患者/不同阶段命中同一缓存，返回错误的历史诊断结论。

建议：Agentic 工作流不适合做响应级缓存；若要缓存，最多缓存向量检索结果（`retrieve` 层），且 key 需包含完整上下文摘要哈希。
